## 一、概述

平台的 AI 能力依托于 DeepSeek Chat 大语言模型，通过精心设计的 4 个 SYSTEM_PROMPTS 驱动不同阶段的智能任务。每个 Prompt 定义了模型的角色、任务目标和输出格式，是 AI 行为质量的关键控制点。

---

## 二、Prompt 清单

### 2.1 extract — 灾害信息抽取

```
你是一个灾害信息抽取助手。根据灾情描述，提取结构化的灾害信息，
以JSON格式返回。包含：灾害类型、严重程度、影响范围、建议响应级别。
```

**使用场景**: 灾情上报后，AI 自动分析自然语言描述，提取结构化关键信息。

**设计考量**:
- 角色定位为"助手"而非"专家"，因为这是预处理步骤不需要创造性
- 明确要求 JSON 格式输出，便于程序解析存入 `incident.extra_data`
- 4 个必提取字段覆盖灾情核心维度（类型、严重性、范围、响应建议）
- 输入来源：`incident.title + incident.description`

**调用位置**: `agent_runner.py:run_extraction()`

---

### 2.2 generate_plan — AI 方案生成

```
你是一个应急管理专家。根据灾情信息和参考预案，生成一份详细的应急处置方案。
方案需包含：一、灾情概述；二、应急响应等级；三、组织机构与职责；
四、处置措施；五、资源调配方案；六、注意事项。请使用专业、规范的表述。
```

**使用场景**: 指挥员选择灾情后点击"AI 生成方案"，触发完整应急处置方案生成。

**设计考量**:
- 角色定位为"应急管理**专家**"，要求输出专业规范
- 明确 6 段式结构，确保方案覆盖全面（概述→等级→组织→措施→资源→注意事项）
- 上下文注入：通过 RAG 检索匹配的参考预案 + 历史采集事件作为 few-shot 参照
- "请使用专业、规范的表述"是风格约束，防止模型输出过于口语化
- `max_tokens=4096` 确保长方案不被截断

**输入上下文构造** (`ai_engine.py:generate_plan_with_ai`):
```
灾情信息：标题/类型/严重程度/描述/影响人数/位置
参考预案+历史数据：(RAG检索结果)
```

**调用位置**: `ai_engine.py:generate_plan_with_ai()`

---

### 2.3 review_plan — 方案审查

```
你是一个应急方案审查专家。审查给定的应急处置方案，
指出存在的问题和改进建议。
```

**使用场景**: 方案生成后，AI 对方案进行二次审查，发现逻辑漏洞和风险盲点。

**设计考量**:
- 角色定位为"审查**专家**"，与生成角色分离（generate 是专家、review 是审查者）
- 输入是完整的方案文本，输出是改进建议清单
- 这个 prompt 的设计目的是引入"AI 自查"机制，在指挥员人工审批前多一层质量保障

**调用位置**: `agent_runner.py:run_plan_review()`

---

### 2.4 retrieve_keywords — 关键词提取

```
你是一个应急关键词提取助手。根据灾情描述，提取3-5个核心关键词
用于检索应急预案，以逗号分隔。
```

**使用场景**: 将灾情描述转换为检索关键词，用于 RAG 知识库的预案匹配。

**设计考量**:
- 轻量级任务，只提取 3-5 个关键词
- 逗号分隔便于直接 split 后传给 `match_plans()`
- 当前系统 ILIKE 检索使用的是 `query_text.split()` 分词，关键词提取有提升空间（未来可将提取的关键词用于更精准的 ILIKE 匹配）

**调用位置**: RAG 检索优化路径（预留，当前 ILIKE 直接使用描述分词）

## 三、降级机制

当 DeepSeek API 不可用（网络超时、Key 无效、余额不足）时，系统自动降级到本地模板引擎：

| AI 任务 | 正常路径 | 降级路径 |
|------|------|------|
| 方案生成 | DeepSeek generate_plan | `generate_plan_with_template()` 本地组装 |
| 信息抽取 | DeepSeek extract | 跳过抽取，使用原始描述 |
| 方案审查 | DeepSeek review_plan | 跳过审查，依赖人工审批 |

降级模板 (`ai_engine.py:generate_plan_with_template`) 包含：
- 5 种灾种分类标签（地震/洪水/地质灾害/森林火灾/其他）
- 4 级严重程度映射（P1-P4）
- 9 个应急组织机构定义
- 4 阶段处置时间线（0-24h / 24-72h / 72h-14d / 恢复重建）
- 匹配预案 + 历史事件的上下文注入

## 四、设计原则总结

| 原则 | 实践 |
|------|------|
| **角色分离** | 4 个 Prompt 各司其职，不做大杂烩 |
| **结构化输出** | extract 要求 JSON、generate_plan 要求 6 段式 |
| **上下文注入** | RAG 检索结果作为 prompt 的参考材料 |
| **安全降级** | AI 不可用时自动切换本地模板，不中断业务流程 |
| **可观测性** | 每次调用记录 agent_runs + citations，失败可重试 |

---

## 五、未来优化方向

1. **Prompt 模板化**: 将 4 个 prompt 抽到 `.env` 或独立配置文件，方便调优
2. **Few-shot 示例**: 在 generate_plan prompt 中嵌入 1-2 个标准方案作为示例
3. **关键词提取启用**: 当前 ILIKE 用分词，未来可启用 retrieve_keywords + 向量检索
4. **审查自动化**: 结合规则引擎（方案字数、关键章节检测）与 AI 审查形成双重质量保障
5. **多模型支持**: prompt 设计为模型无关，可切换到其他 LLM 只需改 `deepseek_client.base_url`
